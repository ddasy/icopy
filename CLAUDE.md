# CLAUDE.md

## 如何在本仓库中导航(MD 索引体系)

**本 MD 体系只有一个职责:当一个请求到来时,让 Claude 以最少的读取次数——快且准——定位到代码,并看清关键点的层级与相关模块的位置。** 它不描述需求,也不记录历史——需求一直在变,而代码是"死"的:它待在固定的位置。MD 只回答两个问题:"X 在哪里"和"X 与什么相关"。任何不服务于定位或关联的内容都不属于这里。

自顶向下,最小作用域优先:

### 第 1 层 —— 全局索引:`/INDEX.md`
**职责:快速定位。** 一份大清单,把每个功能/模块映射到其所在位置。当你不知道某个功能在哪里时从这里进入;当你已经知道工作作用域时跳过。

### 第 2 层 —— 目录 README:`<dir>/README.md`
**职责:逐文件描述与关联。** 每文件一行,外加一个 `## Cross-refs` 小节记录关联。每个含代码的目录都有一个(由下文的同步规则强制)。

### 层级弹性(自伸缩——由 Claude 在日常会话中维护,无需外部工具)

当前层级数不是上限。结构是递归自相似的:**每当任一节点的子项增长到"一跳定位"失效时,就在该节点插入一个 INDEX 层。**

- **加一层**:某个目录 README 超过约 30 条,或某个子系统长出了多个功能域(定位一个功能需通读整个 README)→ 在该目录创建 `INDEX.md`(按功能聚合 → 指向子 README);README 退化为只索引本目录自身的文件。本 monorepo 中,某个 `Packages/<包>` 长大后即按此在该包内插入 `INDEX.md`。
- **减一层**:某个子系统缩小到一眼可览(≤ 约 10 条)→ 把该 INDEX 合并回父级。
- **不变量**(任何加/减层都不得破坏):
  1. 一次定位 = 自顶向下,每层至多读一个文件;
  2. 每个文件只回答"X 在哪里"和"X 与什么相关";
  3. 新层级遵守与既有层级相同的风格约束与同步规则;
  4. 父级索引在同一次改动中更新其指针。

### Cross-refs(跨域引用)
参与跨切代码的 README 携带一个 `## Cross-refs` 小节:
- **intra** —— 指向同作用域内其他目录的指针。

### 何时回退到 grep / find
- README 指向了文件,但你需要文件内的某个具体符号。
- 你在搜索一个索引尚未捕获的跨切模式(事后,把发现补进相应的 INDEX)。

若某 README 或 INDEX 与代码不符,以代码为准——在同一次改动中修正文档。

## 架构基线(目录结构标准,强制)

项目的目录结构是标准化的,而非野蛮生长。基线如下。**新代码必须落在基线之内;要偏离基线,先改基线(这是一次架构决策——在同一次改动中更新本小节 + 受影响的 INDEX/README),再写代码:**

```
icopy/
├── App/                  应用入口与生命周期;AppKit 桥接(NSApplication、菜单栏 StatusItem、全局热键),编排为主
├── Packages/             本地 SwiftPM 包(monorepo 各单元),每个子目录一个 Package.swift;同层内部形态保持一致
│   ├── Core/             领域模型与业务逻辑,无 UI、无平台依赖(剪贴板条目、规则)
│   ├── Clipboard/        剪贴板监听与采集(NSPasteboard 变更捕获),无业务规则
│   ├── Translation/      翻译服务(LM Studio 本地 LLM 调用,可注入协议),无 UI
│   ├── Storage/          持久化层(历史记录读写、检索),无 UI
│   └── UIComponents/     可复用 SwiftUI 视图组件,纯 UI、无业务语义
├── Features/             SwiftUI 功能模块,一个功能一个目录(编排为主:调服务、绑数据、处理事件)
├── Resources/            资源:Assets、Info.plist、entitlements、本地化(非代码目录,不索引)
└── Tests/                测试套件:单元/集成测试与 fixtures(粗粒度索引)
```

1. **一个目录 = 一个职责。** README 第一行的职责陈述就是该目录的边界声明;若一行说不清这个目录是干什么的,它就已经混了——拆。
2. **不许有垃圾目录。** 绝不创建或喂养 `misc/`、`temp/`、或无边界膨胀的 `common/`/`helpers/`——每段代码都有确定的归属;若无合适归属,走规则 3。
3. **放置新功能,三步**:(a) 属于某个既有目录的职责 → 放进去;(b) 是一个新职责 → 建目录 + README(第一行声明边界)并更新父级 INDEX;(c) 拿不准 → 先定职责再写代码;"先放这里、以后再挪"被禁止。
4. **同层目录保持同构。** 同一层上的领域目录共享一套内部形态(如每个 `Packages/<包>/` 都带 `Sources/` + `Tests/` + `Package.swift`);新包复制既有包的形态——别发明新形态。
5. **不许越界伸手。** 跨目录依赖走清晰接口(模块对外暴露 public API,import 类型);伸手进另一个包的内部实现文件被禁止;每条跨目录关系都必须出现在两侧 README 的 `Cross-refs` 中。

## 模块化原则(强制)

**优先模块化;能抽出的接口都抽出来。** 写或改代码时,默认把职责拆成可复用单元,而不是内联。模块化是索引体系的前提:一个塞满多种职责的文件无法用 README 一行描述——**"一行写不下"本身就是该文件应当拆分的信号。**

1. **凡可复用即抽出。** 一段逻辑(数据形状、系统 API 调用、解析/格式化、UI 片段)若在——或可能在——不止一处使用,就放进它该在的层;绝不在调用点复制。
2. **遵守分层,各归其位:** 领域模型与业务规则 → `Packages/Core/`;剪贴板采集 → `Packages/Clipboard/`;持久化 → `Packages/Storage/`;可复用 UI → `Packages/UIComponents/`;功能视图编排 → `Features/<name>/`;应用入口与 AppKit 桥接 → `App/`。
3. **入口/视图/路由文件保持薄。** 只做编排(调服务、绑数据、处理事件);业务规则、解析、数据形状定义沉进其模块。
4. **小而命名良好的接口胜过大杂烩函数。** 拿不准时就拆——给一个抽出的单元命名,远比日后拆解一个内联块便宜。
5. **不做投机式预抽象。** 三行相似代码可以容忍;抽取的触发条件是真实复用或清晰的职责边界,而非想象中的未来。

报告任务完成前,扫一遍 diff:坐在入口/视图/全局状态文件里、本应属于模块层的逻辑,先挪走。启发式:若一个文件的 README 一行描述里出现"以及/并且/还负责",它多半需要拆。

## 强制:代码修改后重新打包并重启 App

每次修改代码后,必须在报告任务完成前退出当前应用进程,重新编译并打包 `.app`,再启动打包后的 App。

1. **先退出旧进程。** 修改 `.swift`、`Package.swift` 或构建相关文件后,先关闭当前 `iCopy` / `icopy` 应用进程。
2. **重新编译并打包。** 运行 `./package.sh`,用最新代码生成 `build/iCopy.app`;涉及测试覆盖或行为风险时先运行 `swift test`。
3. **重启打包 App。** 运行 `open build/iCopy.app`,并确认新进程来自 `build/iCopy.app/Contents/MacOS/iCopy`。
4. **失败必须报告。** 退出进程、编译、测试、打包或重启失败时,在最终回复中说明失败命令与阻塞原因。

## 强制:逐目录 README.md 同步

每个含代码目录的 `README.md` 都是源码的一部分——不是可选文档,而是必须与代码保持同步的契约。每当你在一个目录中增、删、改名、或实质性改动一个文件,就在同一次改动中更新该目录的 README。

1. **每个含代码的子目录都必须有 `README.md`。** 代码 = `.swift`(及少量 `.m`/`.h`);纯产物目录(`.build/`、`DerivedData/`、`build/`、`*.xcodeproj/`、`*.xcworkspace/`、`.swiftpm/`、`Pods/`)豁免。**非人写代码**(生成代码、第三方 vendored 代码、git 子模块)不逐文件索引——在父 README 中一行说明("生成代码,来源是 X,勿手改");**测试/fixture 目录**粗粒度索引(每套件一行),测试 ↔ 被测代码的映射记在被测目录 README 的 Cross-refs 里。
2. **缺 README 的代码目录在任务结束前补上。** 格式:一行职责 + 文件/子目录的项目符号列表,每项一行。
3. **目录内文件被改动 → 在同一次编辑中更新该目录 README**:加了文件 → 加一行;删/改名 → 删/改那一行;职责变了 → 更新描述;纯内部重构、对外不可见 → 不更新。
4. **作用域 = 被改文件自身所在的目录**,而非其父级。仅当目录自身结构变了(增/删子文件夹)才更新父级 README。
5. **条目从简。** README 是索引,不是手册——每文件一行;细节归代码注释或设计文档。

报告任务完成前,对每个被触及的文件核验:(a) 其目录 README 反映当前文件集与角色;(b) `Cross-refs` 仍与代码相符。

**镜像同步**:本文件为其他 agent 备有一个镜像(`CLAUDE.md` ↔ `AGENTS.md`)。二者双向——你此刻正在编辑的那个文件就是本次改动的源;每次编辑后,在同一次改动中重新生成另一个文件,逐字相同,仅头部声明行不同(`# CLAUDE.md` vs `# AGENTS.md`,以及本"镜像同步"条款中对自身文件名的引用)。绝不让二者漂移;若发现失同步,以最近编辑的那个为源覆盖另一个。

### 风格约束(强制)

README / INDEX 只描述**当前状态**——绝不写历史。违反下列任一项即视为任务未完成:

1. **不写日期。** 没有 `2026-06-05`,没有"六月迁移"。变更时间戳归 git log。(被引用文件名中的日期除外。)
2. **不写变更叙事。** 没有"从 X 迁移而来""此前/原先是 Y""Z 已删除/移除""第二阶段完成"。删掉的东西不出现;改过的东西只以当前形态出现。变更理由归提交信息;阶段进度归设计文档。
3. **教训转为现在时约束。** 不讲故事("我们用过 X,崩了,换成 Y")——只留规则("用 Y;绝不用 X——它在 Z 上抛异常")。一条约束 ≤ 1 行。
4. **每文件一行,≤ 120 字符**(中日韩字符按双倍计)。写不下,说明你在写手册——把细节推进代码注释或设计文档;README 只留"这个文件负责什么 + 谁来找它"。
5. **状态标记只出现在 INDEX 的功能表里**(如 "(planned)"、"(skeleton)");目录 README 绝不携带进度。
6. **INDEX 顶部不堆叙事。** 当前状态一句话;活的机制以现在时写进其相应小节——没有阶段号,没有日期。
